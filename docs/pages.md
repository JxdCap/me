# 页面与空间架构

本项目采用单页应用内的空间切换模型。页面不通过传统跳转制造割裂感，而是在同一空间中完成进入、聚焦、阅读和返回。

## 1. 空间层级

### Content Stage

主内容层，由 `HomePage` 承载。

包含：

- `Header`：顶部 Liquid Glass 功能岛。
- `Hero`：头像、标题和技能标签。
- `StillAlive`：动态记录堆叠。
- `Footer`：低优先级页面锚点。

这一层的职责是展示身份和最近内容。它不应该被复杂控件打断。

### Functional Overlay

功能覆盖层，用于临时任务和全局导航。

包含：

- `Index Overlay`：全局菜单。
- 阅读器中的 `reader-bar`：关闭按钮、当前记录上下文、轻量进度边缘。

该层使用 Liquid Glass。它浮在内容上，提供操作，不成为内容本身。

### Reading Surface

阅读层，由 `ZineReader` 承载。

打开卡片后，主页退后并降低存在感，阅读器接管视窗。用户在这里连续浏览记录、图片和正文。

阅读器不是弹窗，也不是卡片详情页。它是一个临时的阅读空间，完成后返回主页。

## 2. 状态模型

### Home

默认状态。

用户看到身份、技能入口和最新动态。顶部功能岛保持可用，但视觉上轻于正文标题。

### Skill Focus

点击技能标签后进入轻聚焦状态。

行为：

- 当前技能高亮。
- 其他技能标签降低权重。
- 说明文字作为内容层展开。
- Footer 隐去，给正文留出空间。

该状态不改变页面层级，只改变局部焦点。

### Index

点击菜单按钮进入全局导航状态。

行为：

- 主内容层后退、模糊、降低透明度。
- Index 覆盖层出现。
- 菜单项依次进入，但动效应短、清楚、可预测。

Index 用于导航，不用于展示内容详情。

### Reader

点击动态卡片进入阅读状态。

行为：

- 主内容层退后。
- 阅读器从当前空间展开。
- 顶部 `reader-bar` 以 Liquid Glass 形式浮在内容之上。
- `reader-bar` 显示当前记录上下文，并随滚动更新。
- 滚动后 `reader-bar` 收缩，给内容让位。

Reader 是唯一允许原生纵向滚动的主空间。

## 3. 路由策略

当前阶段保持单路由首页：

```text
/ -> HomePage
```

菜单中的 `about`、`works`、`notes` 可以先作为未来路由保留，但不应在视觉上暗示已经完成。

未来扩展时，优先使用同一套空间模型：

- 一级内容可以进入新的 content stage。
- 临时任务使用 functional overlay。
- 长内容使用 reading surface。

## 4. 组件职责

### `Header`

职责：

- 菜单入口。
- 明暗主题切换。
- 个人入口预留。

设计要求：

- 整体是一个或多个 Liquid Glass control cluster。
- 按钮不单独漂浮成一组无关胶囊。
- 激活态要明确，但不使用刺眼装饰色。

### `Hero`

职责：

- 建立个人身份。
- 给用户一个明确的第一阅读焦点。
- 提供技能筛选/说明入口。

设计要求：

- 保持内容层属性，不使用厚玻璃。
- 标题应像系统大标题，而不是海报字。

### `StillAlive`

职责：

- 展示最近动态。
- 提供进入阅读流的入口。
- 支持上滑切换。

设计要求：

- 卡片是内容表面，不是控制表面。
- 交互动效保留物理感，但视觉变形克制。

### `ZineReader`

职责：

- 连续阅读所有动态记录。
- 展示图片与正文。
- 提供可见但退让的退出路径。

设计要求：

- 内容滚动优先。
- `reader-bar` 属于功能层，内容记录属于内容层。
- 不使用每篇记录自己的 sticky header。
- 关闭后焦点回到触发阅读器的 `StillAlive` 入口。

## 5. 未来管理入口

未来可以加入私密管理能力，但不应破坏主页的内容优先结构。

建议方向：

- 通过个人入口进入管理面板，而不是隐藏式谜题入口。
- 管理面板作为 functional overlay 或独立 authenticated stage。
- 编辑状态清楚标识，不与公开浏览状态混淆。
- 手机端发布流要优先服务即时记录：文字、地点、图片、发布。
